# 速通新领域（Domain Explorer）

帮用户快速建立对一个陌生领域的全景认知：**历史演化主线 + 当前竞争格局 + 专家共识与争议 + 公开实践信号**，并对照专家与从业者认知的差异。

## 核心理念

1. 理解一个领域的关键是**每个东西是为了解决什么问题而出现的**。叙事主线：问题 → 解决方案 → 后续问题或分化（如有）→ 新解决方案 → …
2. **专家材料和公开实践讨论分开采集、最后对照**，避免互相污染。

## 主流程

1. **确认领域和深度**：用可用的提问方式让用户选择深度（速览 / 入门 / 深入，执行差异见文末分级表）。领域词过泛、有同名歧义、或含易碰撞的数字/型号时，同一回合一并澄清；非交互场景按最合理解释执行并在产出中标注假设。<!-- web-research-standards:A@2026-07-07 -->
2. **领域适配判断**：
   - 领域受产业化 / 监管 / 资本 / 标准 / 地缘强影响（如 AI、医药、能源、金融）→ 在对应研究线中追加该视角
   - 无活跃的垂直从业者社区 → 转向 Reddit、知乎、Quora 等全内容社区搜该领域的对应问题和讨论；泛社区样本代表性更弱，相关判断置信度相应下调并标注
   - 专家与从业者高度重叠（如纯数学）→ 差异对照改为"理论前沿 vs 教学/应用共识"
3. **研究**：按「采集规范」执行四线研究。有并行子代理能力则四线并行（研究子代理可用较经济的模型档），否则串行。
4. **综合**：以历史线为骨架构建演进链，现状线补充当前节点；做"专家 vs 公开实践"差异对照（见采集规范四分类）；当两线冲突且双方证据都不弱、或关键认知证据太弱时，带着具体冲突点追加 1-3 次定向搜索验证，不直接下结论。
5. **产出认知地图**：板块结构见「数据结构与呈现」，渲染方式见「渲染层」。

## 采集规范

### 四条研究线

- **A 历史演化线**：从起源到现在的演化链。每个关键节点回答**解决了什么问题**；如果确实存在，再记录付出的代价 / 引入的新问题——没有就不写，不要硬凑。
- **B 竞争格局线**：当前主要玩家 / 流派 / 路线。每个记录核心主张、代表作、与其他路线的差异维度（如性能、成本、哲学、生态位）。**必须标注时间窗口**（"截至 YYYY-MM"）。
- **C 专家共识与争议线**：综述论文、领域权威的访谈/演讲/辩论、会议 panel、教科书定论。产出**共识清单**（已被视为定论的）和**争议清单**（每条标注双方立场、代表人物、各自依据）。
- **D 公开实践信号线**：搜索 Reddit、Hacker News、知乎、V2EX、垂直论坛/Discord 公开归档、技术博客评论区，找**广受认同的实践认知**：高赞经验帖、踩坑复盘、反直觉结论、与教科书叙述有出入的一线判断。每条记录平台、热度证据、发帖时间、发帖人身份（仅记录公开自述或可验证的职业背景，不可验证时标"身份未知"，不做推断）。**独立探索，不带 C 线的专家框架去搜**。注意保持平台多样性（不同平台人群不同，高赞 ≠ 行业共识）。**输出必须声明样本偏差**：这只是公开可见的实践样本，大量真实从业讨论发生在闭源渠道（内部群、私有社区、会议）。

### 证据分级

每条关键判断按此记录：

```
结论 | 来源类型 | 证据强度（强/中/弱）| 时间范围 | 反例或争议 | 置信度（高/中/低）
```

| 等级 | 判定标准 |
|---|---|
| 强 | 综述/标准/权威教材，或多篇独立高质量来源一致，且时间窗口清楚 |
| 中 | 单一权威来源；或多条社区/行业来源相互印证 |
| 弱 | 单篇博客、孤立高赞帖、厂商材料、未经交叉验证的说法 |

不同来源**不同权**：厂商材料默认降一级；孤立高赞帖默认"弱"，跨平台多条独立讨论相互印证方可升"中"。置信度参考**评论区反应和其他帖子对同一认知的讨论**：多处独立印证升，有力反驳或大量分歧降。评论区最高赞（尤其是反驳）重于帖子热度，作为校正信号核查而非直接当共识；当事人一手发言重于第三方转述；用钱和行为投票的证据（市场赔率、实际迁移、提交记录）在判断采用与实践类问题时可升一级。关键认知尽量保留一条带署名的原话引用。<!-- web-research-standards:B@2026-07-07 -->

### 差异对照四分类

把 D 线高赞认知逐条与 C 线对照：

- **相互印证**：实践经验与专家共识一致
- **实践 tacit knowledge**：从业者知道但专家文献少提的工程细节、隐性成本、工具链真相
- **民间迷思**：实践中流行但专家认为错误或过时（给出专家侧依据）
- **实践已投票**：专家仍在争议、但实践已用脚投票形成事实标准

每条标注**证据等级**和**适用边界**（大厂/创业/开源/学术/地区/周期——很多"实践真相"只在某类组织或某个时代成立）。不确定的宁可标"待验证"，不硬归类。专家与从业者高度重叠的领域（如纯数学），此四分类替换为主流程第 2 步所述的"理论前沿 vs 教学/应用共识"对照。

## 数据结构与呈现

内容板块：演进地图、竞争格局、共识与争议、专家 vs 实践差异、推荐阅读路径、关键术语速查（最易困惑的 3-5 个术语一句话解释）。

```
{
  nodes: [{
    id, label, era,
    problemSolved: string,   // 解决了什么问题
    cost: string | null,     // 代价/新问题（如有；没有为 null）
    keyInsight: string,      // 一句话核心洞察
    type: "origin" | "breakthrough" | "paradigm" | "application" | "frontier",
    year: string | null
  }],
  edges: [{ from, to,
    relation: "led_to" | "enabled" | "solved" | "applied_in" | "evolved_into" | "competes_with",
    label }],
  metadata: {
    domain, depth, asOf,                 // asOf = 信息时间窗口
    keyFigures: [{name, contribution}],
    landscape: [{player, thesis, differentiators, representative}],
    consensus: [{claim, evidenceLevel}],
    controversies: [{question, sideA, sideB, keyPeople}],
    practitionerInsights: [{insight, category, evidenceLevel, scope, source}],
    frontierTopics, practicalApplications
  }
}
```

呈现要求：演进节点默认显示名称和时代标签，展开后显示解决的问题、代价/新问题（如有）、核心洞察、年份；不同 type 用不同颜色/形状，演进关系用有向连线；顶部有领域标题、一句话总结、时间窗口（asOf），并说明本次实际启用/合并/跳过的研究线和明显缺口；<!-- web-research-standards:C@2026-07-07 -->整体视觉避免 AI 味、清晰可读、体现演进感。

## 渲染层（按环境择一）

- **支持交互产物的环境**：生成可交互知识地图（技术栈遵循当前环境惯例，如 Claude artifact 环境用 React + Tailwind 的 `.jsx`；按 era 分层布局、节点可展开、有向连线、节点多时可缩放拖拽），上述内容板块以 tab 或分区呈现，输出到环境约定的产物位置。
- **纯文本/CLI 环境**：输出 Markdown + Mermaid 图表，板块结构与地图版一致。

## 深度分级

| | 速览（5分钟） | 入门（15分钟） | 深入（30分钟+） |
|---|---|---|---|
| 节点数 | 3-5 | 8-12 | 15-20 |
| 研究线 | A 主干演化草图（标注"未充分验证"）+ B/C 合并 + D 轻量 | 四线全量 | 四线全量 + 适配追加线 |
| D 线深度 | 1-2 个高信号来源，输出"实践侧常见判断"小节并标注"未做充分社区验证" | 多平台采集 | 多平台 + 身份线索核验 |
| 差异对照 | 简要提及 | 四分类完整 | 四分类 + 补查验证 |

## 注意事项

- 所有信息标注来源和时间，关键判断遵守证据分级记录格式
- 中文领域用中文标注，英文领域用英文（或中英混合）
- 证据不足或拿不准时如实标注（"待验证""身份未知""信息有限"），不硬凑不硬归类
