# Domain Explorer — 领域探索器

> 平台兼容：完整功能（React artifact 交互地图）需要支持 artifact 渲染的环境（如 Claude 桌面端）；纯 CLI 环境降级为 Markdown + Mermaid 图表输出。

帮用户快速建立对一个陌生领域的全景认知，输出为交互式可视化知识地图。

## 核心理念

这个 skill 的两个核心假设：

1. **理解一个领域最好的方式不是罗列概念，而是理解"每个东西是为了解决什么问题而被发明的"**。知识地图的叙事主线是：问题 → 解决方案 → 新问题 → 新解决方案 → …
2. **专家共识和从业者认知是两个独立证据源，必须分开采集、显式对照**。模型先验是公开文本的加权平均（偏教科书化共识、tacit knowledge 代表性不足、有时间滞后），只有实际去搜社区，才能把"从业者共识"作为独立证据源与"专家共识"归因对照。

## 工作流程

### 第一步：确认领域和深度

用户给出一个领域后，用当前平台的用户提问工具（如 AskUserQuestion；不可用时直接文本询问）让用户选择深度：

- **速览（5分钟）**：3-5 个核心节点，只展示最关键的演进脉络；从业者线降级为轻量扫描
- **入门（15分钟）**：8-12 个节点，覆盖主要分支、竞争格局、共识/争议；全量四线
- **深入（30分钟+）**：15-20 个节点，包含细分方向、争议双方立场、前沿论文/人物、专家 vs 从业者深度对照；全量四线 + 补查

### 第二步：领域适配判断

派研究子代理之前，主代理先快速判断领域性质（可借助 1-2 次 WebSearch）：

- 领域是否受**产业化 / 监管 / 资本 / 标准组织 / 地缘**强影响（如 AI、医药、能源、金融科技、自动驾驶、医疗、教育）？是 → 在对应研究线中追加该视角，或增派一条专线
- 领域是否有**活跃的垂直从业者社区**？没有 → 转向 Reddit、知乎、Quora 等全内容社区搜该领域的对应问题和讨论，不降级从业者线
- 领域中**专家和从业者是否高度重叠**（如纯数学）？是 → 差异分析改为"理论前沿 vs 教学/应用共识"的对照
- 领域宽泛（如"物理学"）→ 照常执行：宽领域同样有主干脉络，以主干演化为骨架、细分方向作分支节点；可顺带提示用户"想聚焦某个分支可以再说"，但不要求缩小范围

### 第三步：并行子代理研究（四线 fan-out）

按深度分级执行（速览不强制四个完整子代理，见文末分级表）。用当前平台的子代理工具（如 Agent）**并行**派出研究子代理（默认 4 条线，按第二步适配结果增减；工人子代理默认 Sonnet，领域艰深时升档）。子代理使用平台可用的网页搜索/抓取工具（需要时先加载，如通过 ToolSearch）；无子代理能力的环境降级为主代理串行执行各线。

- **A 历史演化线**：领域从起源到现在的演化链。每个关键节点回答：**解决了什么问题**；如果确实存在，再记录**付出了什么代价 / 引入了什么新问题**——很多演化节点没有明显代价或新问题，没有就不写，不要硬凑。搜索方向：`"[领域] history evolution"`、关键转折点的一手叙述、"why did X replace Y"类讨论。
- **B 竞争格局线**：当前主要竞争者 / 流派 / 技术路线。每个玩家记录：核心主张、代表作、与其他路线的差异维度（性能/成本/哲学/生态位/商业模式）。**必须标注时间窗口**（"截至 YYYY-MM 的格局"），格局变化快的领域要注明近期变动。搜索方向：`"[领域] landscape comparison 2025 2026"`、`"X vs Y"`、行业分析报告。
- **C 专家共识与争议线**：搜索综述论文、领域大牛的访谈/演讲/辩论、学术会议 panel、教科书定论。产出两份清单：**主流共识清单**（哪些问题已被认为解决/定论）和**活跃争议清单**（每条争议标注双方立场、代表人物、各自依据）。
- **D 从业者视角线**：搜索 Reddit、Hacker News、知乎、V2EX、领域垂直论坛/Discord 公开归档、技术博客评论区。找**广受认同的从业者认知**：高赞的实践经验帖、踩坑复盘、反直觉结论、与教科书叙述有出入的一线判断。每条记录：**平台、热度证据（赞数/讨论量）、发帖时间、发帖人身份线索**。注意：不同平台人群不同（HN 偏创业/工程、知乎偏中文行业、Reddit 分版差异大），高赞 ≠ 行业共识，采集时保持平台多样性。**独立探索，不要带着 C 线的专家框架去搜**——这条线的价值正是发现专家框架之外的一手实践信号。

**统一输出 schema**：每个子代理的每条关键判断必须包含：

```
结论 | 来源类型（论文/教科书/大牛访谈/行业报告/厂商材料/社区热帖/个人博客）
| 证据强度（强/中/弱）| 时间范围 | 反例或争议 | 置信度（高/中/低）
```

论文综述、厂商白皮书、社区热帖**不能同权处理**——厂商材料默认降一级证据强度，孤立高赞帖最高只算"中"。置信度评估要参考**评论区的反应和其他帖子对同一认知的讨论**：被多处独立来源印证的升，评论区有有力反驳或大量分歧的降。

### 第四步：综合与差异分析

主代理汇总四线结果：

1. **构建演进链知识结构**（见下方数据结构），以 A 线为骨架，B/C 线补充现状节点。
2. **专家 vs 从业者差异分析**：把 D 线的高赞认知逐条与 C 线对照，归入四类：
   - **相互印证**：从业者经验与专家共识一致（列出，增强可信度）
   - **从业者 tacit knowledge**：从业者知道但专家文献少提的工程实践细节、隐性成本、工具链真相
   - **民间迷思**：从业者中流行但专家认为错误或过时的认知（要给出专家侧依据）
   - **实践已投票**：专家仍在争议、但从业者已用脚投票形成事实标准的
   
   每条差异标注**证据等级**（强/中/弱）和**适用边界**（大厂/创业公司/开源社区/学术界/特定地区/特定周期）——很多"从业者真相"只在某类组织或某个时代成立。
3. **Targeted follow-up 补查**：当 C/D 冲突且双方证据都不弱、或某条关键认知证据太弱时，主代理带着具体冲突点追加 1-3 次定向搜索验证，不要直接下结论。

### 第五步：构建知识结构

```
{
  nodes: [
    {
      id: string,
      label: string,           // 概念名称
      era: string,             // 时代/阶段标签
      problemSolved: string,   // 解决了什么问题
      cost: string | null,     // 代价/新问题（如有；没有则为 null，不硬凑）
      keyInsight: string,      // 一句话核心洞察
      type: "origin" | "breakthrough" | "paradigm" | "application" | "frontier",
      year: string | null,
    }
  ],
  edges: [
    { from: string, to: string,
      relation: "led_to" | "enabled" | "solved" | "applied_in" | "evolved_into" | "competes_with",
      label: string }
  ],
  metadata: {
    domain: string,
    depth: "速览" | "入门" | "深入",
    asOf: string,                        // 信息时间窗口
    keyFigures: [{name, contribution}],
    landscape: [{                        // 竞争格局
      player: string, thesis: string, differentiators: [string], representative: string
    }],
    consensus: [{claim, evidenceLevel}],           // 专家共识清单
    controversies: [{question, sideA, sideB, keyPeople}],  // 争议清单
    practitionerInsights: [{                       // 专家 vs 从业者差异
      insight: string,
      category: "相互印证" | "tacit_knowledge" | "民间迷思" | "实践已投票",
      evidenceLevel: "强" | "中" | "弱",
      scope: string,                               // 适用边界
      source: string                               // 平台+热度
    }],
    frontierTopics: [string],
    practicalApplications: [string],
  }
}
```

### 第六步：生成可视化知识地图

创建一个 React (.jsx) artifact，要求：

**板块结构**（tab 或分区）：
1. **演进地图**（主视图）：问题→解决方案演进链
2. **竞争格局**：当前玩家/路线对比（卡片或对比表）
3. **共识与争议**：专家共识清单 + 争议清单（争议展示双方立场）
4. **专家 vs 从业者**：四分类差异对照，每条带证据等级和适用边界标签

**交互设计：**
- 节点可点击，展开显示详细信息（problemSolved、cost、keyInsight）
- 不同 type 的节点用不同颜色/形状区分，演进关系用有向箭头连接
- 支持缩放和拖拽（如果节点较多），有图例说明节点类型

**视觉风格：**
- 避免 AI 味，追求大胆但协调的视觉风格
- 整体布局：从左到右的时间流（或从上到下），体现演进感
- 配色大胆但协调，不要用烂大街的紫色渐变；字体选择有个性（可用 Google Fonts CDN）
- 节点之间的连线要有层次感，不能一团乱麻

**内容呈现：**
- 每个节点默认显示名称和时代标签；点击后展开卡片显示 🎯 解决的问题、⚖️ 代价/新问题（如有）、💡 核心洞察、年份
- 地图顶部有领域标题、一句话总结和信息时间窗口（asOf）

**技术实现：**
- React + Tailwind；节点布局用简单分层算法（按 era 分列）；连线用 SVG 路径；状态管理用 useState
- 不要依赖 D3 做布局（太重），但可以用 D3 画 SVG path

### 第七步：补充说明

在地图下方（或作为地图内的一个 tab），用简洁的文字补充：
- **推荐阅读路径**：如果用户想继续深入，应该先看什么、再看什么
- **关键术语速查**：最容易让新手困惑的 3-5 个术语的一句话解释

## 深度分级执行差异

| | 速览 | 入门 | 深入 |
|---|---|---|---|
| 研究线 | A 全量 + B/C 合并为一线 + D 轻量 | 四线全量 | 四线全量 + 适配追加线 |
| D 线深度 | 1-2 个高信号社区来源，输出"实践侧常见判断"小节，标注"未做充分社区验证" | 多平台采集 | 多平台 + 身份线索核验 |
| 差异分析 | 简要提及 | 四分类完整 | 四分类 + 补查验证 |
| 补查 | 无 | 冲突时 | 冲突或证据弱均触发 |

## 输出要求

二者择一：支持 artifact 渲染的环境输出一个 `.jsx` 文件（到当前 session 的 outputs 目录）作为可交互知识地图；纯 CLI 环境输出 Markdown + Mermaid 图表（板块结构与地图版一致）。

## 注意事项

- 所有信息标注来源和时间；关键判断遵守统一 schema（来源类型/证据强度/时间范围/置信度）
- 中文领域用中文标注，英文领域用英文（或中英混合）
- 如果领域太小众搜不到什么，诚实告知并尽力而为
- 永远以"问题→解决方案"的叙事主线组织演进信息（有代价/新问题时一并呈现），而不是简单罗列
- 从业者线独立并行执行，综合阶段才与专家线对照；不确定的对照结论宁可标"待验证"也不硬归类
